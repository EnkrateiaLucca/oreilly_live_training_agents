0. Why Agents? — The Agent Loop from Scratch
What is an Agent?

An agent = LLM + Tools. The model decides when and how to call tools, gaining access to the real world (search, files, APIs, code execution).

  • LLM: reasoning, planning, language
  • Tools: real-world actions & data
  • Loop: keep going until goal is met
OpenAI Function Calling
  • Define tools as JSON schemas (name, description, parameters)
  • Pass schemas to model; model returns tool call objects
  • Your code executes the tool, feeds result back
  • Key: model decides whether and when to call tools
The Manual Agent Loop
Call LLM → Tool call? → Yes → Execute tool → Append result → Call LLM again → No tool call → Return answer

Key insight: AgentExecutor IS a graph under the hood — LangGraph makes this loop explicit and controllable.

1. Intro to LangGraph — Why a Framework?
Why LangGraph over Manual Loops?
  • State management — built-in, typed
  • Cycles — retry and loop natively
  • Conditional routing — branch on any condition
  • Streaming — token-by-token and step-by-step
  • Persistence — checkpoints between steps
  • Observability — LangSmith tracing built-in
LangChain Building Blocks
  • Model access: ChatOpenAI(model="gpt-4o")
  • Prompt templates: reusable strings with dynamic variables
  • Chains: composable LLM + prompt blocks
  • Structured outputs: model.with_structured_output(MyModel)
  • Model + tools: model.bind_tools(tools)
  • Pydantic: validate structured output schemas
2. LangGraph Core Concepts — State · Nodes · Edges
State
Shared data structure (TypedDict or Pydantic model) — the current snapshot of your application. Passed into every node; returned updates are merged back.
Nodes
Python functions that receive the current State as input, perform computation or side-effects, and return a dict of updated State fields.
Edges
Connections between nodes. Fixed edges always go to the same next node. Conditional edges call a routing function to choose dynamically.
Building a Basic Graph
from langgraph.graph import StateGraph, START, END
from typing import TypedDict

class MyState(TypedDict):
    messages: list
    count: int

def my_node(state: MyState) -> dict:
    return {"count": state["count"] + 1}

def router(state: MyState) -> str:
    return "done" if state["count"] >= 3 else "again"

graph = StateGraph(MyState)
graph.add_node("worker", my_node)
graph.add_edge(START, "worker")
graph.add_conditional_edges(
    "worker", router, {"again": "worker", "done": END}
)
app = graph.compile()
MessagesState & Reducers

Use MessagesState for conversation history — it automatically appends messages instead of replacing the list.

from langgraph.graph import MessagesState

class AgentState(MessagesState):
    pass  # add extra fields here

Custom reducers: annotate fields with Annotated[list, add_messages] or your own merge function.

Command Primitive (Post-1.0)

Command combines state update + routing in a single return. Replaces the dict-return + separate conditional edge pattern.

from langgraph.types import Command

def my_node(state):
    return Command(
        update={"result": "done"},
        goto="next_node"  # or END
    )
Multi-Schema State

A graph can have separate schemas for three purposes:

  • Internal state: all fields needed during execution
  • Input schema: what callers provide (input_schema=)
  • Output schema: what the graph returns (output_schema=)

Ask: what do I need to track vs. what do I need to expose?

ReAct Agent Loop
1
Reason
LLM receives state, decides next action or tool to call
2
Act
Tool node executes the tool call from the LLM response
3
Observe
Tool result appended to state messages
4
Repeat
Conditional edge: more tools needed → back to LLM; done → END
3. Local Research Agent (LangGraph + Ollama)
Research Agent Pattern
1 Generate search queries from user question 2 Perform web searches (Tavily, SerpAPI) 3 Summarize search results 4 Reflect — identify knowledge gaps 5 Repeat if gaps remain, else return answer
Key Patterns
  • ConfigurableRunnableConfig: parameterize agents (model choice, depth) without re-compiling the graph
  • Multi-tool prompts: explicit tool descriptions in system prompt improve reliability
  • RAG chunking: recall > precision — a missed chunk can't help; semantic chunking beats fixed-size for retrieval quality
  • Local LLMs: use Ollama for offline / private data
4. LangGraph Post-1.0 Features
Functional API (@entrypoint / @task)

Write graphs as regular Python functions using decorators — no explicit node/edge wiring. LangGraph 0.3+

from langgraph.func import entrypoint, task

@task
def call_model(messages):
    return llm.invoke(messages)

@entrypoint()
def my_graph(state):
    result = call_model(state["messages"]).result()
    return {"messages": result}
MCP Integration

Model Context Protocol — connect agents to any MCP server. Install: pip install langchain-mcp-adapters

from langchain_mcp_adapters.client import (
    MultiServerMCPClient
)
client = MultiServerMCPClient({
    "filesystem": {
        "command": "npx",
        "args": ["-y",
          "@modelcontextprotocol/server-filesystem",
          "/path/to/dir"]
    }
})
tools = await client.get_tools()
Durable Execution
  • LangGraph Platform provides resume-on-failure: if a run is interrupted (network error, timeout, restart), it resumes from the last checkpoint
  • Built on the same checkpointer system (see Notebook 5.0)
  • No extra code required — configure in langgraph.json
  • Critical for long-running agents in production
Subgraphs (overview)
  • Compile a StateGraph → use it as a node in another graph
  • Foundation for multi-agent architectures
  • Each agent/team is self-contained; parent orchestrates
  • Full coverage in Notebook 6.0
5. Persistence & Checkpointers
Checkpointer Options
Checkpointer Use Case Install Notes
MemorySaver Dev / testing Built-in Lost on process exit. Never use in prod.
SqliteSaver Local dev, single-node langgraph-checkpoint-sqlite Survives restarts. Good for prototypes.
PostgresSaver Production langgraph-checkpoint-postgres Concurrency, durability, horizontal scale.
Checkpointer Usage Pattern
from langgraph.checkpoint.memory import MemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver

# Compile with checkpointer
checkpointer = MemorySaver()
app = graph.compile(checkpointer=checkpointer)

# ALWAYS pass thread_id in config
config = {"configurable": {"thread_id": "user-123"}}

# Same thread_id = same conversation
app.invoke({"messages": [...]}, config=config)

# Inspect state
state = app.get_state(config)
history = list(app.get_state_history(config))

# Override state manually
app.update_state(config, {"messages": [correction]})
Cross-Thread Store

Checkpointers are scoped to a single thread. For data shared across conversations use the Store abstraction:

  • User profiles, learned preferences
  • Shared knowledge across sessions
  • InMemoryStore (dev) or external DB (prod)
from langgraph.store.memory import InMemoryStore
store = InMemoryStore()
app = graph.compile(
    checkpointer=checkpointer,
    store=store
)
6. Subgraphs — Multi-Agent Composition
Pattern 1 — Shared State Keys

When the subgraph's schema shares keys with the parent, add the compiled subgraph directly as a node. LangGraph auto-passes matching keys in and merges output keys back.

sub = subgraph.compile()
# Add directly — shared keys handled auto
parent.add_node("agent", sub)

Simplest and most common pattern.

Pattern 2 — Different Schemas

When schemas differ, wrap the subgraph in a node function that handles translation both ways.

sub = subgraph.compile()

def agent_node(state: ParentState):
    # 1. Translate parent → subgraph
    sub_input = {"query": state["input"]}
    # 2. Invoke subgraph
    result = sub.invoke(sub_input)
    # 3. Translate output → parent
    return {"output": result["answer"]}

parent.add_node("agent", agent_node)
Persistence in Subgraphs
  • Compile the parent graph with a checkpointer
  • All subgraph invocations within that thread are automatically checkpointed — no extra wiring
  • Per-invocation persistence (subgraph has its own checkpointer) is useful for isolated agents
When to Use Subgraphs
  • Multi-agent teams — each agent is a subgraph; parent orchestrates
  • Reusable modules — package once, embed in many parents
  • Failure isolation — a crash in a subgraph doesn't kill the parent
  • Different state schemas — specialized agents with their own state

Key APIs — Quick Reference
Graph Construction
from langgraph.graph import (
    StateGraph, MessagesState, START, END
)
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import Command

# Build
graph = StateGraph(MyState)
graph.add_node("node_a", fn_a)
graph.add_node("node_b", fn_b)
graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")
graph.add_conditional_edges(
    "node_b", router,
    {"yes": "node_a", "no": END}
)
app = graph.compile(
    checkpointer=MemorySaver()
)
Invocation & Inspection
# Config with thread_id (required for checkpointing)
config = {"configurable": {"thread_id": "t1"}}

# Invoke (blocking)
result = app.invoke({"messages": [...]}, config)

# Stream steps
for chunk in app.stream(input, config):
    print(chunk)

# Stream tokens
for token in app.astream_events(input, config):
    print(token)

# Inspect / edit state
state = app.get_state(config)
history = list(app.get_state_history(config))
app.update_state(config, {"key": "value"})

# Command return (node function)
return Command(update={"x": 1}, goto="next")

Model + Tools Setup
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from pydantic import BaseModel

@tool
def my_tool(query: str) -> str:
    """Search for information."""
    return search(query)

# Bind tools to model
model = ChatOpenAI(model="gpt-4o")
model_with_tools = model.bind_tools([my_tool])

# Structured output
class MyOutput(BaseModel):
    answer: str
    confidence: float

structured = model.with_structured_output(MyOutput)
